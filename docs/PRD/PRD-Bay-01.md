## 1. 概述

开发一个 Python SDK，让 Yarbo 机器人用户能够便捷地将个人设备数据接入第三方智能家居平台（如 HomeAssistant）或自研平台。

## 2. 背景

Yarbo 产品线包含割草机器人、扫雪机器人等多种户外机器人。当前用户只能通过 Yarbo 官方 App 管理设备，缺乏开放的编程接口。为扩大生态覆盖、满足开发者和高级用户的集成需求，需要提供一个公开发布的 Python SDK，通过封装云端 REST API 和 MQTT 实时通信能力，让用户安全地访问自己的设备数据并进行控制。

### 业务驱动
- 扩展智能家居生态集成（HomeAssistant、自研平台等）
- 提升高级用户和开发者体验
- 建立开放的设备接入标准

### 技术驱动
- 统一设备通信协议封装
- 安全的认证和权限控制机制
- 标准化的多设备类型管理

## 3. 用户故事

- 作为一个 **Yarbo 机器人拥有者**，我希望通过 `pip install` 安装 SDK，以便在我的 Python 项目中快速集成 Yarbo 设备管理能力
- 作为一个 **智能家居开发者**，我希望通过 SDK 获取我账号下的设备列表和实时状态，以便将 Yarbo 设备接入 HomeAssistant 等平台
- 作为一个 **智能家居开发者**，我希望通过 SDK 订阅设备 MQTT 实时推送，以便在自己的平台上实时展示设备状态变化
- 作为一个 **Yarbo 机器人拥有者**，我希望 SDK 确保我只能访问自己的设备数据，以便我的设备信息安全不被泄露

## 4. 功能需求

### 4.1 认证与登录

- **描述**: 用户通过用户名和密码登录，SDK 内部使用 RSA 公钥加密密码，调用云端中转层登录接口获取 token 和 refresh_token
- **输入**: 用户名（邮箱）、密码（明文，SDK 内部加密）
- **输出**: 登录成功后 SDK 内部持有 token 和 refresh_token
- **业务规则**:
  - 密码必须使用 RSA 公钥加密后再传输
  - RSA 公钥通过构造参数传入或从云端公开接口获取
  - 登录失败需返回明确的错误信息（凭证错误、网络异常等）
  - 登录成功后 token 存储在内存中，不落盘

### 4.2 Token 自动刷新

- **描述**: 当 token 过期时，SDK 自动使用 refresh_token 调用云端刷新接口获取新 token，对用户透明
- **输入**: 过期的 token、有效的 refresh_token
- **输出**: 新的 token
- **业务规则**:
  - 调用 REST API 时检测到 401 响应，自动触发 token 刷新
  - 刷新成功后自动重试原请求
  - refresh_token 也过期时，抛出异常提示用户重新登录
  - 刷新过程中的并发请求需要正确处理（避免多次刷新）

### 4.3 设备发现

- **描述**: 获取当前登录用户账号下绑定的所有机器人设备列表
- **输入**: 无（使用当前 token 鉴权）
- **输出**: 设备列表，每个设备包含设备 ID（SN）、设备类型、设备名称等基本信息
- **业务规则**:
  - 只返回当前用户拥有的设备（由云端控制）
  - 支持多种设备类型（割草机、扫雪机等）

### 4.4 MQTT 实时状态订阅

- **描述**: 通过 MQTT 连接订阅设备状态实时推送，用户需要明确指定要订阅的 topic
- **输入**: 用户传入要订阅的 topic（可通过设备能力注册表查询可用 topic）
- **输出**: 通过回调函数接收实时状态更新消息
- **业务规则**:
  - MQTT 使用 token 作为 username 进行 JWT 鉴权（EMQX JWT AUTH, from=username）
  - 用户传入要订阅的 topic，SDK 不做 topic 格式校验或限制（不支持的 topic 由 EMQX 云端 ACL 鉴权拒绝）
  - 用户只能订阅自己设备的 topic（由 EMQX ACL 通过 JWT 中的 acl claim 控制）
  - 支持断线自动重连
  - 支持用户注册自定义回调函数处理消息

### 4.5 REST API 公共层

- **描述**: 所有 REST API 请求的公共基础层，统一处理 token 注入、错误处理、重试等
- **输入**: API 路径、请求参数
- **输出**: API 响应数据
- **业务规则**:
  - 所有请求自动携带 Authorization header
  - 统一的错误响应处理（网络错误、HTTP 错误码、业务错误码）
  - 401 响应自动触发 token 刷新并重试
  - API endpoints 集中管理在单独文件中

### 4.6 设备能力注册表

- **描述**: SDK 内置一个设备能力注册表（device registry），集中声明每种设备类型支持的能力，包括可查询的数据、对应的 MQTT topic 和 REST API 接口，供用户查阅和 SDK 内部使用
- **输入**: 设备类型标识
- **输出**: 该设备类型支持的能力清单（可订阅的 topic 列表、可调用的 REST API 列表、状态字段定义）
- **业务规则**:
  - 注册表集中管理在单独文件中（如 `device_registry.py`）
  - 每种设备类型声明: 支持的 MQTT topic 模板、支持的 REST API 端点、状态字段定义、控制指令定义
  - 用户可通过 SDK 查询某个设备类型支持哪些 topic 和 API
  - 新增设备类型只需在注册表中添加条目，无需修改 SDK 核心代码
  - 本阶段搭建注册表框架并填入基础定义，具体控制指令下阶段补充

### 4.7 SDK 配置管理

- **描述**: SDK 通过构造参数或云端公开接口获取运行所需配置（API URL、MQTT 地址、RSA 公钥等）
- **输入**: 可选的 api_base_url、mqtt_host、mqtt_port、rsa_public_key 等构造参数
- **输出**: 配置好的 SDK 客户端实例
- **业务规则**:
  - 优先级：构造参数 > 云端公开接口返回的配置
  - 传入构造参数则直接使用，未传入的配置项从云端公开接口获取
  - 云端配置接口无需鉴权（配置内容非敏感信息）
  - 云端配置在 SDK 生命周期内只拉取一次（内存缓存）
  - config.py 仅保留 SDK 内部常量（超时时间、重试次数等），不存放环境相关配置
  - SDK 提供 token / refresh_token 属性供宿主应用读取，便于宿主应用自行持久化
  - SDK 提供 restore_session(token, refresh_token) 方法，供宿主应用恢复会话（无需重新登录）

## 5. 非功能需求

- **性能**: MQTT 消息推送延迟不应由 SDK 引入额外显著延迟（< 100ms 处理时间）
- **安全**:
  - 密码传输前必须 RSA 加密
  - Token 仅存储在内存中，不持久化
  - SDK 源码不包含任何密钥或敏感配置
  - MQTT ACL 由云端 JWT claim 控制，SDK 侧不做绕过
- **兼容性**:
  - Python >= 3.10
  - 支持 Linux、macOS、Windows
  - 同步接口（本阶段不提供 async）
- **可维护性**:
  - Topic 集中管理在单独文件
  - API Endpoint 集中管理在单独文件
  - 设备类型定义可扩展
- **发布**:
  - 通过 PyPI 发布，`pip install yarbo-robot-sdk`
  - 源码公开在 GitHub

## 6. 验收标准

- [ ] AC1: 用户可通过 `pip install yarbo-robot-sdk` 安装 SDK

- [ ] AC3: Token 过期后自动刷新，用户无感知
- [ ] AC4: 登录后可获取当前用户的设备列表
- [ ] AC5: 用户可明确指定 topic 进行 MQTT 订阅，并通过回调接收消息
- [ ] AC6: MQTT 连接使用 token 作为 username 进行 JWT 鉴权
- [ ] AC7: MQTT 断线后自动重连
- [ ] AC8: SDK 支持通过构造参数传入配置，未传入时从云端公开接口获取
- [ ] AC12: SDK 提供 token/refresh_token 属性和 restore_session 方法，支持宿主应用持久化和恢复会话
- [ ] AC9: SDK 内置设备能力注册表，用户可查询设备类型支持的 topic 和 API

- [ ] AC11: SDK 有完善的错误处理，登录失败、网络异常等场景有明确的异常类型

## 7. 影响范围

| 模块 | 影响 | 说明 |
|------|------|------|
| SDK 包结构 | 全新创建 | `yarbo_robot_sdk` Python package |
| 云端中转层 | 需新建 | 提供登录、设备列表、设备状态等 API（不在本 SDK 范围内，但 SDK 依赖） |
| EMQX 配置 | 需配置 | JWT AUTH + ACL（不在本 SDK 范围内，但 SDK 依赖） |

## 8. 风险与约束

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 云端中转层 API 尚未开发 | 高 | SDK 无法进行真实联调测试 | SDK 先基于 API 契约开发，使用 mock 测试；中转层 API 设计纳入 ADR |
| EMQX JWT AUTH 配置变更 | 低 | MQTT 鉴权失败 | SDK 预留配置灵活性，鉴权方式可调 |
| 不同设备类型的状态字段尚未完全定义 | 中 | 类型定义可能需要后续调整 | 本阶段搭建类型定义框架，具体字段预留扩展能力 |
| PyPI 包名冲突 | 低 | 无法使用预期包名发布 | 提前检查包名可用性 |

## 9. 阶段规划

| 阶段 | 内容 | 说明 |
|------|------|------|
| **本阶段（Phase 1）** | 基础建设 | 认证登录、REST API 封装、MQTT 初始化与订阅、设备发现、设备能力注册表框架 |
| 下阶段（Phase 2） | 业务实现 | 设备状态查询（MQTT）、设备控制指令（开始/暂停/恢复/回充等） |
